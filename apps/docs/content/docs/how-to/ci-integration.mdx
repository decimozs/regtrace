Regtrace produces exit codes and machine-readable output, making it a natural
fit for CI/CD pipelines.

## Basic CI setup

<Steps>
  <Step>
    ### Download the binary in your pipeline

    <Callout type="info">
      Regtrace is a standalone binary — no runtime dependencies, no npm install
      needed.
    </Callout>

    <Tabs items={['GitHub Actions', 'GitLab CI', 'Generic']}>
      <Tab value="GitHub Actions">
        ```yaml
        - name: Download Regtrace
          run: |
            curl -L -o /usr/local/bin/regtrace https://github.com/decimozs/regtrace/releases/latest/download/regtrace
            chmod +x /usr/local/bin/regtrace
        ```
      </Tab>
      <Tab value="GitLab CI">
        ```yaml
        before_script:
          - curl -L -o /usr/local/bin/regtrace https://github.com/decimozs/regtrace/releases/latest/download/regtrace
          - chmod +x /usr/local/bin/regtrace
        ```
      </Tab>
      <Tab value="Generic">
        Any CI platform can download the binary:

        ```bash
        curl -L -o /usr/local/bin/regtrace https://github.com/decimozs/regtrace/releases/latest/download/regtrace
        chmod +x /usr/local/bin/regtrace
        ```
      </Tab>
    </Tabs>
  </Step>
  <Step>
    ### Run evaluation

    ```bash
    regtrace run --ci
    ```

    The `--ci` flag suppresses color, prints a CI-mode notice, and exits with
    code 1 if any quality gate fails.
  </Step>
  <Step>
    ### Check exit code

    | Code | Meaning |
    |------|---------|
    | 0 | All quality gates passed |
    | 1 | One or more quality gates failed |
    | 2 | Config or schema error — evaluation did not run |

    Exit code 1 allows the pipeline to reject the change. Exit code 2
    indicates a setup problem that needs attention.
  </Step>
</Steps>

## Complete GitHub Actions workflow

```yaml
name: Evaluate LLM

on:
  push:
    branches: [main]
  pull_request:

jobs:
  evaluate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Download Regtrace binary
        run: |
          curl -L -o /usr/local/bin/regtrace https://github.com/decimozs/regtrace/releases/latest/download/regtrace
          chmod +x /usr/local/bin/regtrace

      - name: Run evaluation
        run: regtrace run --ci
        env:
          GROQ_API_KEY: ${{ secrets.GROQ_API_KEY }}
```

## CI mode

The `--ci` flag:

- Suppresses color output
- Exits with code 1 if any suite fails quality gates

```bash
regtrace run --ci
```

## Auto-detection

Regtrace auto-detects CI environments. When `--ci` is not explicitly
passed, it checks for `CI`, `GITHUB_ACTIONS`, `GITLAB_CI`, or `CIRCLECI`
environment variables and suppresses color automatically. Use `--no-ci` to
override auto-detection in CI environments.

## Disabling CI auto-detection

```yaml
output:
  ci_mode_auto_detect: false
```

## JSON output for pipelines

For programmatic consumption:

```bash
regtrace run --format json
```

JSON output is written to stdout, enabling piping directly to `jq`:

```bash
regtrace run --format json | jq '.suite.suite_score'
```

## Fail-fast with `--bail`

Stop at the first suite that fails quality gates:

```bash
regtrace run --ci --bail
```

This saves pipeline time when multiple golden sets are configured and an early
failure is sufficient to reject the change.

## Quality gates as pipeline gates

Configure quality gates in `regtrace.config.yaml`:

```yaml
quality_gates:
  suite_score_minimum: 0.7
  max_failed_test_cases: 0
  max_low_confidence_ratio: 0.1
  regression_gate: true
```

These map directly to pipeline pass/fail. No separate pipeline logic needed.

## Markdown reports in PR comments

Generate a Markdown report and post it in a PR:

```bash
regtrace run --format markdown --output report.md
```

Then use a GitHub Action like `actions/github-script` to read and comment
with the report content.
